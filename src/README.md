# SIMD scanner core

Hand-written assembly sources for the relocation-free scanner blob
that ships in `dist/`. Not included in the published npm package —
this directory only matters if you're modifying the inner loops.

## Build

```sh
make            # emits ../lib/scanner/bytes-<arch>.ts
make test       # native + Rosetta correctness/perf tests
make dump       # otool disassembly of arm64 build
make clean
```

The pipeline:

1. `clang -arch <a> -c scan_<a>.S` → `.o`
2. Wrapped in a dylib (for symbol-offset extraction; not shipped)
3. `extract_text.sh` pulls the raw `__text` bytes via `otool -l`
4. `emit_ts.sh` writes a TypeScript module with the bytes as a
   `Uint8Array` plus a `SYMBOLS` map of byte offsets per entry point,
   straight into `../lib/scanner/bytes-<arch>.ts`

After `make`, run `npm run build` at the package root to recompile
the TypeScript surface against the regenerated bytes.

## Design

### Self-contained machine code

The four entry points (`_scan`, `_scan1`, `_scan_parallel`,
`_scan1_parallel`) live in a single `.S` file per architecture so
inter-function references resolve to PC-relative `E8` call (x86_64)
or `bl` (arm64) at assembly time — no relocations. `nm -u` and `otool
-r` on the resulting dylib both report empty.

External symbols (`dispatch_apply_f`, the queue handle) are passed as
function-pointer arguments rather than referenced by name, so dyld
never needs to fix up anything. The raw `__text` bytes are
position-independent and runnable from any executable page.

### Parallelism

GCD's `dispatch_apply_f` drives the workers. The worker function
lives in the same `__text` section as the leaves and is reached via
PC-relative `lea ... [rip + sym]` (x86_64) / `adr` (arm64), so the
blob stays relocation-free.

### Two leaves

- `_scan` — up to 4 targets in SSE / NEON registers, compared in
  parallel against 2 u64 slots per iteration. Scalar fallback for
  >4 targets.
- `_scan1` — single-target fast path, unrolled to 4 slots per
  iteration. ~5× faster single-threaded than the general `_scan`
  because the inner loop is dominated by per-batch overhead, not
  memory bandwidth.

The parallel wrappers (`_scan_parallel`, `_scan1_parallel`) build a
ctx struct on the stack and `dispatch_apply_f` over `nthreads`
chunks; each worker calls the leaf with its slice. Workers write
their results into per-thread output sub-buffers so there's no
cross-thread coordination.

### Hit recording

The leaves always increment the running hit count, but only store to
`out` when `count < out_cap`. The return value is the total hit count
(possibly > `out_cap`), so the caller can detect truncation and re-run
with a larger buffer.

## Performance

Measured on an Apple M5 with 8 logical cores, 512 MB buffer:

| variant         | x86_64 single (Rosetta) | x86_64 ×8 | arm64 single | arm64 ×8 |
|-----------------|------------------------:|----------:|-------------:|---------:|
| `scan` (4 tgt)  | 9 GB/s                  | 101 GB/s  | 13 GB/s      | 98 GB/s  |
| `scan1` (1 tgt) | 52 GB/s                 | 111 GB/s  | 65 GB/s      | 116 GB/s |

`scan1` parallel is DRAM-bound (~110 GB/s ceiling on this hardware);
single-thread is CPU-bound.

## Entry-point signatures

All four are SysV AMD64 (x86_64) / AAPCS64 (arm64) regular C
functions. Hit indices are u32 (sufficient for any practical chunk
size).

```c
uint64_t scan(
    const uint64_t *buf,      // 16-byte aligned, readable
    uint64_t        qcount,   // number of u64 slots
    uint64_t        mask,
    const uint64_t *targets,  // pre-masked
    uint64_t        ntargets, // 1..N
    uint32_t       *out,
    uint64_t        out_cap);

uint64_t scan1(
    const uint64_t *buf,
    uint64_t        qcount,
    uint64_t        mask,
    uint64_t        target,   // pre-masked
    uint32_t       *out,
    uint64_t        out_cap);

void scan_parallel(
    void (*dispatch_apply_f)(size_t, void*, void*, void(*)(void*, size_t)),
    void           *queue,
    const uint64_t *buf,
    uint64_t        qcount,
    uint64_t        mask,
    const uint64_t *targets,
    uint64_t        ntargets,
    uint32_t       *out_flat,       // nthreads * out_cap_each
    uint64_t        out_cap_each,
    uint64_t       *out_counts,     // nthreads u64s
    uint64_t       *out_chunk_offs, // nthreads u64s; per-thread starting slot
    uint64_t        nthreads);

void scan1_parallel(
    void (*dispatch_apply_f)(size_t, void*, void*, void(*)(void*, size_t)),
    void           *queue,
    const uint64_t *buf,
    uint64_t        qcount,
    uint64_t        mask,
    uint64_t        target,         // pre-masked
    uint32_t       *out_flat,
    uint64_t        out_cap_each,
    uint64_t       *out_counts,
    uint64_t       *out_chunk_offs,
    uint64_t        nthreads);
```

Hit indices written by the parallel variants are LOCAL to each
thread's chunk — add `out_chunk_offs[i]` to recover absolute slot
indices. The TypeScript `Scanner` class handles this.

## Genericness

Fits naturally:

- arm64e ISA / vtable scans (with PAC-stripping mask)
- Plain pointer hunts (`mask = ~0`)
- Tagged-pointer matching (any mask)
- Magic-number / sentinel scans at 8-byte alignment
- Inode / GUID searches that fit in 8 bytes

Doesn't fit:

- Sub-8-byte granularity (string / byte patterns)
- Unaligned matches
- Substring / regex / Boyer-Moore
- Range queries (`min ≤ value ≤ max`)
- Multi-slot patterns ("field A == X *and* field B == Y")
- Hundreds of targets (scalar fallback is O(N) per slot)

## Future variants

Each would be a new symbol in the same blob; the build pipeline
auto-emits the offset in `SYMBOLS`.

- **`scan_pair`** — two adjacent u64 slots both match `(targetA,
  targetB)`. Eliminates most false positives when an object's isa or
  vtable happens to appear as a bare value elsewhere.
- **`scan_range`** — `low ≤ value ≤ high`. NEON `cmhs`/`cmhi` pair.
- **`scan_aligned_u32`** — 4-byte stride; doubles throughput when the
  target fits in 32 bits.
- **AVX2 `_scan1`** — 4 u64 lanes per ymm, 8 u64 / iter. Requires
  macOS 15+ for Rosetta; native Intel since 2013.

## Implementation notes

### Why `adr` / `lea [rip+sym]` for the worker pointer

`bl <local>` and `call <local>` resolve at assemble time to
PC-relative encodings, so the bytes are valid no matter where the
blob is copied. Same for `adr` / `lea ...[rip + worker]` to take the
address of the worker function. Together these are the only intra-blob
references; no relocation entries are emitted.

### Alignment

The leaves load 16 bytes at a time via `movdqu` (x86_64) / `ldr q`
(arm64). `movdqu` is unaligned-safe; `ldr q` requires only 4-byte
alignment.

### Hit count overflow

Hit count is `uint64_t`; out indices are `uint32_t`. A 32 GB buffer
has 4 G u64 slots — at the edge of `uint32_t`. If you ever scan a
single buffer near that size, return slot index as u64.
